# RestroBooth — Benchmark Specification

**Status:** Phase 0 deliverable. **These run in Phase 1, before any feature is built on them.**
**Last updated:** 2026-07-13

Phase 0 writes no application code, so two decisions ([ADR-0006](adr/0006-override-resolution.md) and the RLS mechanism in [TENANCY.md](TENANCY.md) §4) are made on reasoning and marked **provisional**. This document makes them falsifiable: it specifies exactly what Phase 1 must measure, against what data, and what the pass/fail thresholds are.

**Rule: Phase 1's first task is to run these. If either fails, we take the escalation path *before* Phase 2 builds on the assumption.** The one thing we will not do is discover this in Phase 9.

## Shared fixture

Both benchmarks run against the same synthetic dataset, generated by `packages/db/bench/seed.ts`.

| Dimension | Value | Why |
|---|---|---|
| Orgs | 2 | Franchisee isolation must be in the data, not just in the tests |
| Brands | 4 (2 per org) | The cloud-kitchen case must be exercised |
| Outlets | **20** | The brief's stated chain scale |
| Stores | 28 | Includes one outlet running 4 brands (the cloud kitchen) |
| Menu items | **200 per brand** = 800 | |
| Channels | **6** | `dinein`, `zomato`, `swiggy`, `ondc`, `direct`, `captain` |
| Dayparts | 4 | breakfast, lunch, happy_hour, dinner |
| Active promos | 10 | |
| **Override rows** | **~15% of cells** | ⚠️ **Realistically sparse.** A dense set would unfairly favour materialisation and give us the wrong answer. |
| Users | 60 | incl. 4 cluster managers with overlapping outlet groups |
| **`order_items`** | **9 M** | one year at chain scale ([ERD.md](ERD.md) §6) |
| `orders` | 2.2 M | |
| `bills` | 2.2 M | |
| Partitions | 12 monthly | Must exercise partition pruning |

Run on a **Supabase Pro-equivalent instance** (not Free — Free's 500 MB cannot hold this fixture, which is itself a finding, and it is the one in [ADR-0001](adr/0001-hosting.md)).

Each benchmark: **200 runs, report p50 / p95 / p99, and always `EXPLAIN (ANALYZE, BUFFERS)` the p95 case.** A number without a plan is not a result — the plan is what tells us *why*, and the why is what determines the fix.

## BENCH-01 — RLS: does the membership lookup hold?

**The question** (brief §10.5): does the `SECURITY DEFINER` membership lookup hold up at 20 outlets and 9 M rows, or do accessible-outlet IDs need to be cached in the JWT?

**What is actually being tested:** whether Postgres hoists `accessible_outlet_ids()` into a once-per-statement InitPlan, or calls it once per row. That is the difference between 40 ms and a timeout, and it is the single most common way a Supabase RLS deployment falls over.

### Query set

Run each as each of four roles: `cashier@outlet`, `outlet_manager`, `cluster_manager` (group of 5 outlets), `org_owner` (all 20). **The cluster manager is the interesting one** — its outlet set is neither a single row nor everything, so it is the case an index cannot trivially serve.

| # | Query | Threshold (p95) |
|---|---|---|
| Q1 | **POS hot path:** open order + items for one table session | **< 20 ms** |
| Q2 | **POS menu load:** resolve full menu for a store (also feeds BENCH-02) | **< 50 ms** |
| Q3 | **KDS:** active KOTs for an outlet, last 4 h | **< 30 ms** |
| Q4 | **Floor map:** all table sessions for an outlet | **< 20 ms** |
| Q5 | **Bill finalise:** insert bill + tax lines + payment, in a transaction | **< 100 ms** |
| Q6 | **Report:** day-end summary for one outlet, one business_date | **< 200 ms** |
| Q7 | **Cluster report:** 30-day summary across 5 outlets (from rollups) | **< 500 ms** |
| Q8 | **Adversarial:** the full 15-case suite from [TENANCY.md](TENANCY.md) §6 | correctness, not latency — **all must deny** |

**Overall pass: every query meets its threshold with RLS enabled, and the `EXPLAIN` shows `accessible_outlet_ids()` evaluated ONCE (InitPlan), not per-row.**

### The comparison that makes the result meaningful

Run the whole set **three ways** and compare — a single number in isolation tells us nothing:

| Variant | Purpose |
|---|---|
| **A. RLS off** | The floor. This is the cost of the query itself. |
| **B. RLS on, `STABLE` + `(select …)` wrapper** | **The proposed design.** |
| **C. RLS on, naive (no wrapper, `VOLATILE`)** | The trap. Expected to be catastrophically slow — **and demonstrating that is itself a deliverable**, because it is what stops someone "simplifying" the wrapper away in six months. |

If **B is within ~2× of A**, the design holds and we ship it.

### If BENCH-01 fails — escalation, cheapest first

1. **Index `memberships (user_id)` and `outlet_group_members (outlet_id)`.** Almost certainly already the issue if the function is slow in isolation.
2. **Verify the InitPlan hoist.** If `EXPLAIN` shows per-row evaluation, the wrapper or the `STABLE` marking is wrong. **Fix this before concluding anything else** — everything downstream is meaningless if the function runs 9 M times.
3. **Materialise `user_accessible_outlets (user_id, outlet_id)`** as a real table, maintained by trigger on `memberships` / `outlet_group_members`. Turns the function into a plain indexed join. **Most likely landing spot if the function form is too slow.**
4. **Cache the outlet-ID array in the JWT** (`app_metadata.outlet_ids`), refreshed on membership change. Fastest possible — zero DB work in the policy.
   ⚠️ **But it has a real security cost, and it is the reason this is last:** a JWT is valid until it expires. **Revoking a membership does not revoke an outstanding token.** A fired manager keeps access until their JWT expires. Mitigations (short TTL, a token-version claim checked against the DB) claw back much of the performance win — which is precisely why this is the *last* resort and not the first.

## BENCH-02 — Override resolution: live or materialised?

**The question** (brief §10.2, [ADR-0006](adr/0006-override-resolution.md)): does live resolution hold at 20 outlets × 200 items × 6 channels?

### Query set

| # | Query | Threshold (p95) |
|---|---|---|
| R1 | Resolve **full 200-item menu** for `(store, channel=dinein, now)` | **< 50 ms** ← the gate |
| R2 | Resolve full menu for `(store, channel=zomato, now)` — dense overrides | **< 50 ms** |
| R3 | Resolve full menu **during an active daypart + 2 active promos** (worst case) | **< 80 ms** |
| R4 | Resolve **one item** (the cart / upsell path) | **< 5 ms** |
| R5 | **Channel menu push:** resolve all 6 channels × all 28 stores | **< 10 s** (batch, not interactive) |
| R6 | Resolve at a **future timestamp** (a scheduled price change) | **< 50 ms** — proves effective-dating costs nothing |

**Overall pass: R1 p95 < 50 ms.** That is the gate, because R1 sits inside the Booth's 2 s LCP budget and inside the POS's 100 ms interaction budget.

### Correctness gate — runs regardless of the timing result

**The 21-row precedence table in [TENANCY.md](TENANCY.md) §7.4 must pass, exhaustively, before any timing number is even reported.** A fast resolver that prices a dish wrong is worthless. Rows **8, 12, 17, and 21** are the ones a naive implementation gets wrong — in particular **row 17** (an 86 must not erase a price override, because price and availability resolve independently).

### If BENCH-02 fails — escalation, cheapest first

Per [ADR-0006](adr/0006-override-resolution.md). We do **not** jump to a materialised view:

1. **Partial index** on `(menu_item_id, status, specificity desc) where status='published'`, plus a `distinct on` formulation.
2. **Application cache** keyed on `(store, channel, daypart, promo_set_hash, menu_version)`, invalidated by Postgres `NOTIFY` on publish/86. **Most likely landing spot.** Preserves every advantage of live resolution — effective-dating still needs no cron, 86 is still instant — and gets a ~5 ms hot read.
3. **Materialised per-store view, refreshed on publish** — only if 1 and 2 both fail. If we land here we must *also* build refresh-reliability infrastructure (including an alarm for "a scheduled price change did not fire at midnight"), which live resolution gave us for free. **Budget for that; it is not a free swap.**

## Reporting

**A provisional ADR that is still provisional at the end of Phase 1 is a bug in the process, not a pending task.**
