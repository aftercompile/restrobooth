# ADR-0003 — ORM: Drizzle

**Status:** Accepted (confirming the brief, with one caveat)
**Date:** 2026-07-13

## Context

We need schema definition, migrations, and typed queries against Postgres, from Next.js Route Handlers, Server Actions, and possibly Edge runtimes. The brief specifies Drizzle. This ADR records *why*, and what it costs — because "the brief said so" is not a reason a future maintainer can evaluate.

Options considered: **Drizzle**, **Prisma**, **Kysely**, **raw SQL + pgtyped**, **Supabase's generated client**.

## Decision

**Drizzle ORM, with schema in `packages/db`, migration-first.**

### Why

1. **Migration-first, not push-first.** Standing rule §9.4 is "write the migration; never mutate schema by hand." Drizzle Kit generates SQL migration files that we read, edit, and commit. We will hand-edit them constantly, because the interesting parts of this schema — the `exclude using gist` constraint on invoice blocks, the partial unique index on open business days, generated columns, RLS policies, partitioned tables — are things **no ORM generates.** Drizzle's willingness to get out of the way and let us write SQL is the actual selling point.

2. **It is SQL-shaped.** The override resolver ([TENANCY.md](../TENANCY.md) §7.3) is a single SQL function. The RLS policies are SQL. The rollups are SQL. An ORM that fights SQL would be a liability here; Drizzle's query builder maps 1:1 onto SQL and lets us drop to raw SQL without ceremony.

3. **Edge-compatible, no engine binary.** Prisma's query engine (historically a Rust binary, latterly WASM) is a persistent source of friction on edge runtimes and in serverless cold starts. Drizzle is just TypeScript.

4. **Types come from our schema, not from introspecting the DB.** This matters because the DB has RLS and partitions that an introspector would model badly.

### Why not the alternatives

- **Prisma** — heavier, historically poor at partitioned tables, RLS, and generated columns; its migration system resists hand-edited SQL, which is exactly what we need to do. The edge story has improved but is still more moving parts than we want.
- **Kysely** — genuinely excellent and a close call. It is a better *query builder* than Drizzle. But it has no migration story of its own and no schema-as-source-of-truth, so we would be bolting a migration tool alongside it. Drizzle gives us both in one.
- **Raw SQL + pgtyped** — maximum control, and honestly tempting given how much raw SQL this project will contain. Rejected because the CRUD surface (menu, console, admin) is large and boring, and hand-writing it is a waste of the team's time. **We reserve the right to use raw SQL wherever Drizzle is awkward** — see the caveat.
- **Supabase generated client** — couples us to Supabase, which [ADR-0001](0001-hosting.md) explicitly forbids in the escape-hatch rule.

## The caveat, stated plainly

**Drizzle does not model the parts of this schema that matter most.** Partitioning, RLS policies, `exclude` constraints, generated columns, and SQL functions are all outside its schema DSL. They will live in **hand-written SQL migration files** that Drizzle Kit runs but does not generate, and Drizzle's TS schema will be, in those places, a *description* of a database it did not fully create.

This is an accepted, deliberate split, and the risk it creates is **drift**: the TS schema says one thing, the DB says another, and nobody notices. Mitigations:

1. **CI asserts `drizzle-kit check` finds no drift** against a freshly-migrated database.
2. **The RLS and constraint behaviour is tested against the real database**, not against the TS types. The adversarial suite in [TENANCY.md](../TENANCY.md) §6 runs on real Postgres in CI (Supabase local / Testcontainers). Types cannot prove RLS works; only a real query as a real role can.
3. **`packages/domain` has zero dependency on Drizzle** or on any DB. It takes plain objects and returns plain objects. If we ever replace Drizzle, the money math is untouched — which is the whole reason for that boundary.

## Consequences

- **Positive:** one tool for schema + migrations + queries; edge-friendly; no engine binary; raw SQL always available.
- **Positive:** replacing Drizzle later means rewriting the query layer only — not the domain layer, not the schema (which is SQL), and not the tests.
- **Negative:** the TS schema is not the whole truth of the database, and CI must be the thing that keeps them honest. Accept and enforce.
